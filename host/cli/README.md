# host/cli

## 1. 定位

`host/cli/` 是 `industrial-network-planner` 的第一宿主形态。

它的职责不是承载复杂业务系统，而是提供一个最小、独立、可移植的命令行入口，用来验证：

- `core/` 是否可脱离旧系统独立工作
- 标准输入包是否足以驱动方案生成
- 输出文档装配链路是否能够形成最小闭环

CLI 在体系中的角色应保持轻量：

- 负责接收输入
- 负责做最小必要校验
- 负责调用核心规则与模板
- 负责把结果写入输出文件

它不负责：

- 承担 adapter 的来源系统逻辑
- 改写核心方法规则
- 承担 Web 式交互体验
- 提前引入复杂运行时依赖

## 2. MVP 目标

CLI 第一版只需要完成一个最小闭环：

1. 读取一个标准输入 JSON
2. 校验输入是否满足最小合同要求
3. 加载 `core/` 中的核心规则与模板
4. 组装出一份客户方案 Markdown
5. 将结果写入指定输出文件或 `output/`

MVP 的成功标准不是“功能很多”，而是：

- 能独立运行
- 输入边界清晰
- 输出结果稳定
- 后续容易扩展为 Web / API 宿主

## 3. CLI 最小命令形态

建议第一版先只提供一个主命令：

```bash
planner generate --input <input.json> --output <output.md>
```

可接受的最小形式：

```bash
planner generate --input core/examples/standard-input-example.json
```

当未显式指定 `--output` 时，建议自动输出到：

```text
output/<project-slug>-solution.md
```

其中 `project-slug` 可从输入中的项目名称推导；若无法推导，则使用时间戳或固定默认名。

## 4. 输入约定

CLI 第一版应只接受“标准输入包”，而不直接接受任何来源系统专用结构。

输入合同来源：

- `core/references/input-contract.md`
- `core/references/portable-input-package-spec.md`

示例输入：

- `core/examples/standard-input-example.json`

这意味着：

- 来源系统数据转换属于 `adapters/` 范畴
- CLI 不应内置当前系统页面字段映射逻辑
- CLI 只面向标准化后的输入运行

## 5. 输出约定

第一版输出统一为 Markdown。

推荐输出内容包括：

- 项目背景与目标
- 需求与约束摘要
- 网络设计方案正文
- 设备/分区/连接等关键设计结果
- 假设项与待确认项
- 风险与后续建议

输出骨架应以以下资产为主：

- `core/templates/customer-solution-template.md`
- `core/references/report-outline.md`
- `core/references/document-assembly-rules.md`

## 6. 推荐模块边界

CLI MVP 建议按以下模块边界拆分，而不是把逻辑堆进一个入口脚本：

### 6.1 入口层

职责：

- 解析命令行参数
- 组织执行流程
- 处理退出码

建议能力：

- `generate` 子命令
- `--input`
- `--output`
- `--stdout`（可选）
- `--strict`（可选）

### 6.2 输入读取模块

职责：

- 读取 JSON 文件
- 解析内容
- 处理文件不存在、JSON 非法等错误

### 6.3 标准输入校验模块

职责：

- 检查最小必需字段
- 检查结构合法性
- 输出清晰的校验错误
- 区分“阻断错误”和“可保留为假设项的缺口”

### 6.4 核心资产加载模块

职责：

- 加载 `core/` 中的模板和规则性文档
- 形成生成过程所需的最小上下文对象

第一版可以先做“文件装配型加载”，不急于实现复杂规则引擎。

### 6.5 文档装配模块

职责：

- 根据输入数据和核心规则组织章节
- 生成 Markdown 文本
- 将缺失项显式标记为假设项/待确认项

### 6.6 输出模块

职责：

- 决定输出路径
- 写入 Markdown 文件
- 确保输出目录存在
- 输出生成摘要

## 7. 建议执行流程

建议第一版 `generate` 的流程如下：

1. 解析 CLI 参数
2. 读取输入 JSON
3. 进行标准输入校验
4. 加载模板与规则文档
5. 生成结构化中间结果
6. 装配 Markdown 文档
7. 写入输出文件
8. 返回执行结果与摘要

## 8. 输入校验策略

建议把输入问题分为两类：

### 8.1 阻断错误

遇到以下情况应直接失败：

- 输入文件不存在
- JSON 语法错误
- 根结构不是预期对象
- 缺失关键身份字段或项目基本信息到无法生成最小文档

### 8.2 非阻断缺口

遇到以下情况可继续生成，但必须在输出中显式标记：

- 部分设计参数缺失
- 部分现场约束不明确
- 设备清单不完整
- 某些结论只能基于经验假设

这类缺口应沉淀为：

- 假设项
- 待确认项
- 风险提示

## 9. 退出码建议

CLI 第一版建议使用明确退出码：

- `0`：成功生成
- `1`：通用执行失败
- `2`：参数错误
- `3`：输入读取失败
- `4`：输入校验失败
- `5`：输出写入失败

## 10. 与 adapter 的边界

CLI 可以在未来支持：

```bash
planner generate --adapter current-framework --input runtime-export.json
```

但这不应进入第一版 MVP。

第一版应保持：

- CLI 仅接收标准输入包
- adapter 转换独立存在于 `adapters/`
- 不让宿主层吞掉适配层职责

## 11. 下一步建议

在本目录下继续推进时，建议顺序为：

1. 先明确 `cli-mvp-plan.md`
2. 再补 `interface-contract.md`（如有需要）
3. 再决定实现语言与目录布局
4. 最后进入第一版命令行实现
