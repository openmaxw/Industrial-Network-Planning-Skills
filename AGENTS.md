
# 当前环境下的稳定执行策略

背景说明：
当前请求链路为：
Codex → DAVINCI Azure Gateway → Azure OpenAI Service → OpenAI。

该链路在以下场景中更容易出现不稳定：
- 长时间流式输出
- 单轮中包含较多步骤的执行型任务
- 长篇说明文本与工具调用混合在同一回合
- 超长回复在完成前发生 reconnect、中断或未完成回滚

因此，助手在该环境下必须默认采用“稳定性优先”的执行方式，而不是单轮最大化输出。

## 核心原则
优先控制单轮输出长度和单轮任务复杂度。
对于较大任务，应自动拆分为多个阶段连续完成，而不是默认在一个超长回合中全部完成。
除非确有必要，否则不要要求用户对每一个小步骤逐一确认。

## 执行规则

1. 文档任务
- 自动将长文档任务拆成 3–4 个逻辑部分连续完成。
- 默认连续执行，不要每一部分都停下来等待用户确认。
- 不要默认一次性流式输出整篇长文档。
- 优先采用分块写文件、分块追加的方式产出文档。
- 每完成一块，只做一句简短进度更新。
- 全部完成后，只返回简短完成说明和文件路径。

2. 代码任务
- 自动将较大的代码修改拆成若干批次执行。
- 在没有真实决策分叉的前提下，按批次连续完成。
- 避免把“大量分析 + 长解释 + 多文件修改 + 全量验证”全部塞进同一个超长回合。
- 每一批完成后，只做一句简短进度更新。

3. 分析任务
- 优先使用本地工具进行读取、搜索、归纳，再生成分析结论。
- 先本地读取和分析，再自动分块输出结果。
- 不要默认一次性输出超长分析全文。
- 如任务较大，应先给结构化结论，再逐步展开。

4. 文件产出任务
- 自动分块写入文件，避免一次性生成过长内容后再统一落盘。
- 能分段写入时，优先分段写入。
- 文件写完后，只返回简短状态和文件路径。
- 避免“先长篇解释，再开始写文件”的模式。

5. 交互方式
- 不要在每个微小步骤后都停下来征求确认。
- 默认自动分解任务并连续执行。
- 只有在以下情况才中断询问用户：
  - 存在真实决策分叉
  - 需要权限提升或审批
  - 涉及高风险或破坏性操作
  - 存在无法安全假设的关键歧义

6. 输出方式
- 优先短回执、短总结、阶段性完成说明。
- 尽量避免“长篇 commentary + 紧接工具调用”的模式。
- 如任务较大，应分阶段完成，而不是一次性长回合完成。
- 能短就短，能分块就分块，能落文件就优先落文件。

## 默认工作模式

1. 对于仓库理解类任务
- 先本地读取
- 再短总结
- 如需长分析，拆块输出

2. 对于长文档任务
- 先拆结构
- 再分块写入
- 每块结束短回执
- 最终返回文件路径

3. 对于代码整改任务
- 先拆批次
- 分批改动
- 每批次后做简短进度更新
- 再做验证

4. 对于交付/报告任务
- 先构造结构化内容
- 再分块写入文件
- 避免一次性全文流式输出

## 实际策略偏好
- 输入可以相对较大，但输出必须尽量受控。
- 单轮任务越大，越应拆块。
- 优先使用本地工具完成能本地完成的部分。
- 只有必须由模型生成的部分，才交给模型输出。
- 生成内容较长时，优先写文件，不优先长流式展示。

## 目标
在当前链路能力下，最大化稳定性、完成率和结果保留率，减少长流式输出失败、reconnect、未完成消息回滚等问题，同时尽量保持助手的自动化执行能力。

## 回复风格偏好
- 默认采用“精准、简短、不重复”的回复方式。
- 优先先给结论，再补必要信息；无必要时不展开背景说明。
- 避免重复用户已知信息、避免同义反复、避免为了完整而堆叠表述。
- 当用户未要求详细解释时，控制回复长度，只保留对当前决策有帮助的信息。
- 当任务已经通过文件交付后，聊天中只做简短状态说明和关键路径提示，不重复文件内容。
- 如果存在更详细但非必要的信息，默认省略，除非用户明确要求展开。
