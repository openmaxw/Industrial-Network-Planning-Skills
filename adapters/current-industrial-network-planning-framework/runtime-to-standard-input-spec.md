# 运行时导出到标准输入模型转换规范

本文件定义如何把当前系统导出的运行时数据转换为 `industrial-network-planner` 使用的标准输入模型。

## 1. 目标

将当前系统的：

- `methodologyKey`
- `formData`
- `recordCollections`

转换为统一的标准输入对象，以供 skill 生成客户方案。

## 2. 输入来源

当前系统通过 `handleExportRuntime()` 导出 JSON，主要包含：

- `methodologyKey`
- `exportedAt`
- `formData`
- `recordCollections`

其中：

- `formData` 适合承载标准页字段值与输出页摘要
- `recordCollections` 适合承载列表型采集、推演和迭代记录

## 3. 转换原则

- 先按页面职责识别数据语义，再做字段归并
- 不要求一次性将所有页面字段逐项硬编码
- 优先构造标准输入域，再考虑客户文档章节
- 缺失字段允许为空，但必须进入 `openItems`

## 4. 基本转换流程

### Step 1：识别经验体类型

根据 `methodologyKey` 判断当前导出数据来自：

- ISA95 主线
- IEC62443 主线
- 未来的总经验体组合导出

如果是单主线导出，转换时应保留“当前仅覆盖部分总经验体”的标记。

### Step 2：读取 `recordCollections`

优先从 `recordCollections` 中提取结构化记录，因为这些记录通常更接近原始项目事实和推演结论。

典型来源包括：

- 企业信息
- 站点结构
- 业务流程
- 对象分配
- 设计推演记录
- 特殊判断与经验候选

### Step 3：读取 `formData`

从 `formData` 中提取：

- 标准页字段
- 输出页摘要
- 局部说明文字

注意：

- 输出页摘要只能作为补充来源
- 不得用输出页摘要替代原始记录和推演记录

### Step 4：归并到标准输入域

将识别出的内容归并到以下域：

- `project`
- `survey`
- `currentNetwork`
- `assets`
- `communications`
- `addressing`
- `security`
- `design`
- `openItems`

### Step 5：补足待确认项

当以下情况发生时，应自动写入 `openItems`：

- 某章节关键数据缺失
- 同一事实来源冲突
- 设计判断需要额外现场确认
- 品牌、数量、位置、审计粒度等关键项未闭合

## 5. 域级转换建议

### 5.1 `project`

优先来源：

- 企业信息页
- 站点结构页
- 项目概述类字段

应整理为：

- 项目名称
- 客户名称
- 站点名称
- 项目目标
- 项目范围
- 项目约束

### 5.2 `survey`

优先来源：

- 调研页
- 业务流程页
- 问题记录页
- 输出页中的现状摘要

### 5.3 `currentNetwork`

优先来源：

- 网络拓扑类页面
- 连接/边界/通道类页面
- ISA95 协同结构输出页
- IEC62443 边界设计页

### 5.4 `assets`

优先来源：

- 对象分配
- 设备归属
- 系统记录
- 层级节点与对象关系

### 5.5 `communications`

优先来源：

- 业务流程
- 接入规则相关页面
- 通道设计页
- 远程维护相关页面

### 5.6 `addressing`

优先来源：

- 地址规划规则
- 地址记录页
- 网段/VLAN 相关页
- 设计推演中的地址结论

### 5.7 `security`

优先来源：

- 区域识别页
- 通道设计页
- 边界控制页
- 安全设计页
- 威胁与风险相关页

### 5.8 `design`

优先来源：

- ISA95 协同设计页
- IEC62443 安全设计页
- 设计说明输出页
- 结果落地页

### 5.9 `openItems`

优先来源：

- 特殊判断
- 经验候选
- 风险评估页
- 缺失字段
- 冲突字段

## 6. 转换输出要求

转换后的标准输入对象应满足：

- 可以直接用于渲染客户方案模板
- 各域允许为空，但结构必须稳定
- 所有不确定信息必须显式进入 `openItems`

## 7. 建议的后续实现方式

后续若进入代码实现阶段，建议新增一个“运行时导出 JSON → 标准输入对象”的转换模块，并按本规范实现。
