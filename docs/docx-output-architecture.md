# DOCX 输出架构设计

本文件定义 `industrial-network-planner` 从当前 Markdown / 结构化交付件输出过渡到 Word (`.docx`) 正式交付件的架构方案。

## 目标

在保留当前正式版生成能力的基础上，新增适合客户正式交付的 Word 输出渠道，使文档具备：

- 封面
- 自动目录
- 正文章节
- 页眉页脚
- 页码
- 原生表格
- 嵌入式拓扑图
- 附录表格

## 设计原则

- 不直接从自然语言反向拼 Word
- 先形成结构化中间模型，再输出到 Markdown / DOCX
- 图、表、章节、附录都应可单独寻址与排版
- Markdown 继续保留为快速预览与调试输出
- DOCX 作为正式客户交付版输出

## 中间内容模型

建议建立统一的文档中间模型，最少包含以下对象：

### 1. 文档元数据
- 项目名称
- 客户名称
- 地点
- 文档版本
- 编制日期
- 编制单位

### 2. 章节对象
- 章节标题
- 章节编号
- 章节摘要
- 正文段落
- 图表引用
- 待确认项

### 3. 图对象
- 图编号
- 图标题
- 图类型（总体拓扑图 / 边界拓扑图）
- 图源（Mermaid / 后续 SVG）
- 图注说明

### 4. 表对象
- 表编号
- 表标题
- 表头
- 表体
- 适用说明

### 5. 附录对象
- 附录标题
- 附录类型
- 附录内容

## 输出分层

### 第一层：结构化生成
由当前 planner / renderer 继续负责生成：

- 正式版章节顺序
- 图对象
- 表对象
- 待确认项
- 闭环项

### 第二层：渲染渠道

#### Markdown 渠道
- 用于快速生成与审核
- 保留 Mermaid 代码块
- 表格直接内嵌

#### DOCX 渠道
- 用于客户正式交付
- Mermaid 图需转换为静态图片后嵌入
- 表格转为 Word 原生表格
- 目录由 Word 样式驱动生成

## 推荐的 DOCX 页面结构

### 封面
- 项目名称
- 客户名称
- 文档名称
- 版本号
- 日期

### 修订记录
- 版本
- 日期
- 修订内容
- 编制 / 审核 / 批准

### 目录
- 按 Word 标题样式自动生成

### 正文
按当前“强论证型交付结构”生成：
- 项目概述
- 现场调研与现状结论
- 设计原则与技术路线
- 需求与约束分析
- 技术选择与方案比较
- 总体网络架构方案
- 网络拓扑图
- 网络分区与边界控制方案
- 地址、VLAN 与子网规划
- 关键设备部署方案
- 通信与运维接入方案
- 实施方案与切换策略
- 风险分析与待确认事项
- 实施结论与定版条件

### 附录
- 拓扑图集
- 访问控制矩阵明细
- VLAN / 子网 / IP 明细表
- 设备部署明细表
- 对象与接口清单

## 图输出策略

当前阶段图以 Mermaid 生成逻辑为主。进入 DOCX 时，建议：

- 先将 Mermaid 渲染为 SVG 或 PNG
- 再嵌入 Word
- 每张图配图号与图名

后续可升级为：

- 结构化图模型
- SVG 图纸风格输出
- DOCX 内嵌高分辨率图

## 表输出策略

当前正式版已有以下核心表：

- 访问控制矩阵摘要表
- 地址与 VLAN 定版摘要表
- 关键设备部署定版表
- 定版闭环表

DOCX 输出时建议：

- 转为 Word 原生表格
- 统一表头样式
- 自动编号
- 支持跨页表头重复

## 实施建议顺序

### 阶段 1
定义统一的文档中间模型（章节 / 图 / 表 / 附录）

### 阶段 2
保留 Markdown 输出不变，同时让 renderer 输出中间对象

### 阶段 3
新增 DOCX 渲染器，将中间对象排版为 Word 文档

### 阶段 4
增加图片导出和主题样式模板

## 当前判断

当前系统已具备：
- 强论证型章节结构
- 图表输出链路
- 审批级 / 施工级表格能力

当前已完成：
- 统一中间模型
- DOCX 输出渠道
- 基于本地绘图器的拓扑图导出与嵌入
- 封面、目录、页眉页脚与页码

仍可继续增强：
- 图号/表号自动编号
- 修订记录页
- 更工程化的图纸风格
- 更精细的 Word 版式模板

因此，DOCX 已不再是规划中的下一层，而是当前正式交付件引擎的正式输出层之一。
