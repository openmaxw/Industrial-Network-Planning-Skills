# 客户方案文档装配规则

本文件用于定义 `industrial-network-planner` 在生成客户方案时，如何把多来源内容组织为一份结构一致、证据清晰、边界明确的正式文档。

它解决的核心问题包括：

1. 多个来源的内容如何合并
2. 不同来源之间冲突时如何取舍
3. 事实、建议、假设、待确认项如何组织
4. 哪些内容进入正文，哪些内容进入附录或边界说明

## 1. 装配总原则

- 先确定章节目标，再装配内容
- 先装配事实，再装配建议
- 先装配主结论，再装配解释说明
- 先保留基础事实记录，再进行摘要与收敛表达
- 不得用调研总结替代关键原始调研内容
- 对核心章节，应形成“结论 → 原始输入 → 设计约束 → 设计响应 → 待确认项”的分层结构
- 先处理冲突，再生成正文
- 先区分正文与附录，再进行细节展开

## 2. 文档来源优先级

当同一主题存在多个来源时，按以下优先级取舍：

1. 采集页中的明确项目事实
2. 推演页中的结构化设计结论
3. 判断页中的规则依据与判断结论
4. 输出页中的摘要内容
5. 迭代页中的补充信息

规则说明：

- 输出页不能替代采集页和推演页
- 迭代页不能直接覆盖正式设计结论
- 判断页用于解释“为什么这样设计”，不能替代项目事实本身

## 3. 冲突处理规则

### 3.1 事实冲突

若两个页面对同一项目事实描述不一致：

- 优先采用来源更原始、更新更明确的采集记录
- 若无法判断，以“待确认项”方式输出，不写定稿结论

### 3.2 设计结论冲突

若推演页与输出页摘要不一致：

- 优先采用推演页结论
- 输出页仅作为补充表述参考

### 3.3 规则判断与设计结果冲突

若规则判断建议与最终设计结论存在不一致：

- 优先检查是否存在额外约束或特殊判断
- 若存在，应在正文或边界说明中解释设计偏离原因
- 若不存在充分原因，应优先保留更符合规则判断的表达

### 3.4 缺失信息导致的冲突

若同一章节必须依赖的信息不完整：

- 允许输出原则性建议
- 不允许输出精确定稿细节
- 缺失点必须进入“风险、假设与待确认项”章节

## 4. 正文、附录与内部信息分流规则

### 4.1 进入正文的内容

正文应包含：

- 已确认项目事实
- 已完成推演的核心设计结论
- 对客户有直接决策价值的结构化建议
- 必须说明的设计依据和方法逻辑

### 4.2 进入附录的内容

附录适合承载：

- 资产清单摘要
- IP/VLAN/子网清单
- 连接关系清单
- 术语与缩略语
- 细节性表格和补充说明

### 4.3 不直接进入客户正文的内容

以下内容一般不直接写入客户正文：

- 经验候选
- 内部复盘性文字
- 未经整理的规则库表达
- 未闭合的内部假设草稿

这些内容可用于：

- 风险与待确认项章节
- 内部沉淀资料
- 后续迭代输入

## 5. 章节装配规则

### 5.1 项目概述与建设目标

装配顺序：

1. 项目基本事实
2. 范围与目标
3. 关键约束

禁止：

- 写入未经确认的扩展范围

### 5.2 现状网络与调研结论

装配顺序：

1. 调研结论摘要
2. 现场观察原始记录
3. 调研发现原始记录
4. 调研关注与限制
5. 收敛后的关键设计约束

禁止：

- 用后续目标设计替代现状描述

### 5.3 设计依据与方法说明

装配顺序：

1. 项目输入依据
2. ISA95 + IEC62443 方法主线
3. 关键规则主题影响

禁止：

- 直接复制内部规则库明细作为客户文案

### 5.4 总体网络架构方案

装配顺序：

1. 架构结论
2. 架构与分层原始依据
3. 关键设计约束
4. 设计响应
5. 与现状差异和优化点

禁止：

- 在信息不足时写过度细化的最终部署图结论

### 5.5 ISA95 层级建模与系统协同结构

装配顺序：

1. 业务场景与系统协同摘要
2. 业务场景原始记录
3. 对象归属与角色记录
4. 跨层协同路径
5. 对网络设计形成的直接约束

### 5.6 IEC62443 分区分域与安全边界设计

装配顺序：

1. zone 划分
2. conduit 组织
3. 边界与访问控制原则
4. 远程维护边界要求

### 5.7 网络拓扑与通信路径说明

装配顺序：

1. 主要连接结构
2. 关键通信与访问原始线索
3. 跨域与受控访问路径
4. 设计响应

### 5.8 IP 地址、VLAN 与子网规划

装配顺序：

1. 地址规划结论
2. 地址继承与保留原始条件
3. 关键设计约束
4. 分段/VLAN/子网设计响应
5. 预留与扩展策略

若信息不足：

- 只输出规划原则和建议，不输出精确定稿表

### 5.9 关键设备与部署建议

装配顺序：

1. 设备与部署结论
2. 设备与部署原始条件
3. 关键设计约束
4. 部署响应
5. 实施条件和限制

### 5.10 实施步骤与迁移建议

装配顺序：

1. 前置条件
2. 阶段划分
3. 风险控制点
4. 迁移建议

### 5.11 风险、假设与待确认项

装配顺序：

1. 风险项
2. 假设项
3. 待现场复核项
4. 待客户确认项

要求：

- 必须单列
- 不能零散藏在正文里

## 6. 表达风格规则

### 6.1 事实表达

用于：

- 已确认信息
- 已存在的网络现状
- 已采集对象和边界信息

特点：

- 简洁、明确、可引用

### 6.2 建议表达

用于：

- 目标架构建议
- 地址规划建议
- 安全设计建议
- 实施建议

特点：

- 说明适用条件
- 说明建议依据

### 6.3 假设表达

用于：

- 为保持方案完整而暂时采用的前提

特点：

- 必须显式标注

### 6.4 待确认表达

用于：

- 现场未核实信息
- 客户未确认条件
- 品牌、数量、位置、容量等关键未定项

特点：

- 不得伪装成设计定稿

## 7. 最终检查规则

文档生成前应自检：

1. 每个章节是否都有可回溯来源
2. 是否把输出页当成了唯一依据
3. 是否把建议写成了事实
4. 是否遗漏了关键待确认项
5. 是否把内部复盘内容混入了客户正文

若以上任一项不满足，应调整后再输出。
