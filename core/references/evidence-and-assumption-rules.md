# 证据、假设与待确认项规则

本文件用于约束 skill 的表达边界，避免把建议写成事实，把推测写成结论。

## 1. 证据分类

### 1.1 已确认事实

来源包括：

- 用户明确提供且一致的信息
- 系统中有明确记录的项目事实
- 经核实的现状拓扑、资产、地址、约束、需求信息

表述方式：

- 使用确定性表达
- 可直接写入方案正文

### 1.2 规划建议

来源包括：

- 基于方法体与项目现状形成的设计建议
- 基于现网问题和目标要求形成的优化建议

表述方式：

- 使用建议性表达
- 要说明建议依据与适用前提

### 1.3 假设

来源包括：

- 项目未明确但为继续组织方案必须临时采用的合理前提

表述方式：

- 明确写为假设
- 必须汇总进入风险、假设与待确认项章节

### 1.4 待确认项

来源包括：

- 缺失信息
- 现场待复核项
- 客户未确认项
- 品牌、容量、位置、策略粒度等关键未定项

表述方式：

- 明确写为待确认
- 不得以正文定稿结论替代

## 2. 强制边界

- 不得把建议拓扑写成现网已建成拓扑
- 不得把估算 IP 规划写成已定稿地址表
- 不得把未确认安全设备部署位置写成实施确定方案
- 不得把现场未核实资产数量写成精确清单

## 3. 输出要求

文档正文中的每项关键结论都应能归入以下之一：

- 已确认事实
- 规划建议
- 假设
- 待确认项

若无法归类，说明该内容证据不足，不应写入正文结论。
