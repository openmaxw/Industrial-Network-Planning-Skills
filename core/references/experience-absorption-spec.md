# 经验体吸收规范

本文件定义 `industrial-network-planner` 如何准确、完整地吸收一个经验体中的调研、判断、推演和结果落地过程。

## 1. 目标

本 skill 吸收的不是零散字段，而是一整套经验体中的工作过程。

所谓“吸收经验体”，至少包括以下四层：

1. 吸收资料采集结构
2. 吸收规则判断结构
3. 吸收设计推演结构
4. 吸收结果落地与经验迭代结构

只有同时吸收这四层，skill 才能真正按经验体工作，而不是只做字段拼接。

## 2. 吸收原则

### 2.1 先吸收过程，再吸收字段

字段本身不是核心，字段所在的页面、阶段、判断意图、输出用途才是核心。

因此吸收顺序必须是：

1. 识别经验体的阶段结构
2. 识别每一阶段的目标
3. 识别每一页面承担的决策职责
4. 再识别页面中的字段与记录

### 2.2 先吸收语义，再吸收命名

不同经验体字段名可能不同，但语义职责可能相同。

例如：

- 企业信息 / 项目概况 / 站点基础
- 对象分配 / 设备归属 / 系统归类
- 网络拓扑 / 连接关系 / 边界结构
- 地址规划 / 网段组织 / VLAN 设计

吸收时优先按语义归并到标准输入模型，而不是按字段名逐条硬编码。

### 2.3 先吸收决策链，再吸收结果页

结果页只能作为摘要来源，不能替代调研与判断链本身。

skill 必须优先理解：

- 采集了什么
- 依据什么判断
- 如何完成推演
- 最终如何落地成文

## 3. 四层吸收模型

## 3.1 资料采集层

目标：吸收经验体要求采集哪些信息，以及这些信息支撑什么后续决策。

需要识别：

- 资料采集导航组中有哪些页面
- 每个页面对应哪类对象或场景
- 每个页面中的字段或记录表达了什么项目事实
- 这些事实将支撑哪些判断主题或设计步骤

在当前仓库中，典型采集对象包括：

- 企业与站点结构
- 业务流程
- 对象/设备/系统归属
- 环境条件
- 实施约束
- 网络现状与连接现状

## 3.2 规则判断层

目标：吸收经验体在“如何判断”上的知识结构。

需要识别：

- 启用了哪些规则主题
- 各规则主题解决什么问题
- 哪些规则主题直接影响方案章节
- 哪些规则主题只影响内部判断，不直接出现在最终文档中

当前平台的规则主题包括但不限于：

- 场景规则
- 分层规则
- 区域规则
- 通道规则
- 边界互联规则
- 技术选择规则
- 地址规划规则
- 性能规则
- 稳定性规则
- 冗余规则
- 接入规则
- 选型规则
- 部署规则
- 安全规则

吸收时应建立“规则主题 → 设计决策 → 方案章节”的映射。

## 3.3 设计推演层

目标：吸收经验体如何从事实与规则走到设计结论。

需要识别：

- 设计推演页的主题是什么
- 推演页如何引用前面的采集信息与规则判断
- 推演页中哪些字段是结论、哪些字段是理由、哪些字段是经验迭代入口

例如：

- 协同设计页 → 形成系统协同和部署组织结论
- 安全设计页 → 形成区域、通道、远维边界和保护措施结论

skill 在吸收时必须保留这种“从输入到结论”的链条，而不是只取最后一个结论字段。

## 3.4 结果落地与经验迭代层

目标：吸收经验体如何把推演结果组织为项目输出，以及如何保留经验反馈。

需要识别：

- 输出页有哪些类型
- 每个输出页面向什么对象（客户、内部、复盘、案例沉淀）
- 哪些内容适合进入客户交付文档
- 哪些内容只适合内部复盘或经验回写

同时还需要识别：

- 特殊判断记录
- 经验候选记录
- 迭代总结记录

这些内容通常不直接进入客户主文档，但应进入假设、边界或内部实施说明。

## 4. 经验体吸收后的内部结构

skill 吸收经验体后，应形成如下内部模型：

### 4.1 采集蓝图

定义：

- 必采信息
- 可选信息
- 每类信息的语义角色
- 信息缺失时影响哪个设计步骤

### 4.2 判断蓝图

定义：

- 经验体启用哪些规则主题
- 各主题的判断目标
- 各主题影响哪些设计结论

### 4.3 推演蓝图

定义：

- 设计推演链条
- 各推演页产出什么结论
- 各结论如何进入客户方案章节

### 4.4 输出蓝图

定义：

- 哪些内容用于客户方案
- 哪些内容用于内部复盘
- 哪些内容用于经验沉淀

## 5. 对最终客户文档的作用

经验体吸收完成后，客户方案中的每一章都应能回溯到以下来源之一：

- 资料采集事实
- 规则判断依据
- 设计推演结论
- 输出页摘要
- 假设与待确认项

如果某章节无法回溯来源，说明吸收不完整，不应直接作为定稿方案内容。

## 6. 强制要求

- 不得只吸收输出页，不吸收采集与判断过程
- 不得只吸收字段名，不吸收页面职责
- 不得把经验迭代记录直接当作客户确定性结论
- 必须保留“输入 → 判断 → 推演 → 输出”的完整链路
