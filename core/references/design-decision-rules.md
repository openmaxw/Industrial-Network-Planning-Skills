# 设计与判断规则

本文件用于约束 skill 在网络规划过程中如何做判断、如何表达结论。

## 1. 基本规则

- 先理解现状，再提出目标方案
- 先确认边界，再细化地址和设备建议
- 先识别约束，再形成实施建议
- 先标识不确定项，再生成文档结论

## 2. 优先级规则

当信息出现冲突时，优先级建议如下：

1. 明确的项目事实与现场确认结果
2. 已记录的系统化采集信息
3. 已形成的规则判断结果
4. 已有输出页摘要
5. 推导性判断

若高优先级信息缺失，不得用低优先级猜测替代事实。

## 3. 规划判断规则

### 3.1 分层与分区

- 优先依据对象角色、业务协同与边界要求确定层级与分区
- 不以单一设备名称代替结构性判断
- 跨层协同路径必须在方案中显式说明

### 3.2 冗余与高可用

- 若无明确可靠性目标，不得擅自承诺具体冗余架构
- 可给出冗余建议，但必须标记适用前提和待确认条件

### 3.3 地址规划

- 优先保证清晰边界、可维护性与可扩展性
- 若现网继承限制较强，应说明继承原则与调整边界
- 若地址基础信息不完整，只能给规划原则，不能给精确分配定稿

### 3.4 设备与部署建议

- 仅在已有场景与约束支持下给出设备级建议
- 若缺少品牌、环境、安装和供电条件，不得输出过度细化的设备定稿

### 3.5 实施与迁移

- 实施建议应与停机窗口、割接条件、现网继承约束一致
- 若缺少实施窗口信息，应将实施路径标记为待确认

## 4. 安全设计规则

- zone 与 conduit 划分必须与业务对象、网络边界和访问路径一致
- 远程运维路径必须作为独立控制重点说明
- 对未确认的安全设备部署位置、策略粒度、审计范围，不得写成确定性落地结果

## 5. 成文规则

- 事实、建议、假设、待确认项必须分开表达
- 每章先结论后说明，但不能跳过依据
- 可适度归纳，不可编造依据
